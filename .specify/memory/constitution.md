# pp-zz-birthday19 宪法

## 核心原则

### 一、代码质量（非协商性）

所有代码在合并前**必须**具备可读性、可维护性，且逻辑自明。

具体规则：

- 函数**必须**拥有单一、命名清晰的职责。超过 30 行的函数**必须**拆分，
  除非有经过审批的书面例外说明。
- **禁止**使用魔法数字和无解释的常量；使用具名常量，并附中文内联注释说明原因。
- 死代码在变为无用时**必须**立即删除——**禁止**保留注释掉的代码块。
- 所有公共 API 表面（函数、类、HTTP 端点）**必须**附带中文文档注释，
  描述用途、参数和返回值。
- **禁止**使用 Lombok、MapStruct 等代码自动生成框架；所有代码**必须**手动编写，以保证可读性和可控性。
- 复杂度违规情况**必须**在编写代码之前，在 `plan.md` 的 `Complexity Tracking` 表格中
  记录并提供明确的理由。

**原因**：可读的代码是长期开发效率的首要保障。禁止代码生成框架确保团队对每一行代码负责，避免隐式逻辑。

---

### 二、测试标准（非协商性）

自动化测试是一等交付物，而非事后补充。

- 测试驱动开发（TDD）对所有新功能均为**强制性**：测试**必须**先于任何实现代码编写，
  并确认处于失败（RED）状态。
- 最低覆盖率门控：测试套件的**行覆盖率 ≥ 80%**。覆盖率低于 80% 的 PR **不得**合并。
- 测试分类及要求：
  - **单元测试**（JUnit 5）：**必须**覆盖所有非平凡的纯函数、Service 和工具类方法。
  - **集成测试**（Testcontainers）：**必须**覆盖所有 Repository 层、外部服务调用和 API 端点。
  - **契约测试**：针对 `specs/[###-feature]/contracts/` 中定义的每个公共 API 契约，
    **必须**编写对应的契约测试。
- 测试**必须**具有确定性和隔离性——测试用例之间不得共享可变状态；单元测试中不得有网络调用。
- 不稳定测试**必须**在发现后一个迭代周期内修复或隔离。
- 每个 Bug 修复**必须**先编写复现测试，确认测试失败后再修复，直至测试通过。

**原因**：未经测试的代码是不可维护的代码。80% 门控在严谨性与交付速度之间取得平衡；
契约测试可防止 API 静默破坏。

---

### 三、用户体验一致性

所有面向用户的界面**必须**行为可预测，并符合已建立的交互模式。

- 每条面向用户的错误信息**必须**清晰说明：发生了什么、为什么发生、以及用户**应当**
  采取什么行动。**禁止**直接暴露晦涩的错误码或堆栈跟踪。
- 所有 REST API 端点**必须**遵循统一的响应结构：
  ```json
  {
    "code": "SUCCESS | ERROR_CODE",
    "message": "中文描述",
    "data": {}
  }
  ```
- 导航流程**必须**一致：无论从哪个入口触发，相同操作**必须**产生相同结果。
- 术语**必须**在 UI 标签、文档、错误信息和日志输出中保持一致。当定义三个或更多领域术语时，
  **必须**在 `specs/glossary.md` 中维护词汇表。
- 多端一致性：考虑到未来 PC Web 端、移动 App 端的多端支持，API 设计**必须**保持向前兼容，
  不得在无版本号的情况下修改已有端点的响应结构。
- 无障碍性：所有 Web 界面**必须**符合 WCAG 2.1 AA 标准。

**原因**：不一致性会侵蚀用户信任。统一的 API 响应结构确保多端客户端无缝接入。

---

### 四、性能要求

性能目标是设计约束，而非事后考虑。

- 每个功能规格**必须**在 `plan.md` 的 `Technical Context → Performance Goals` 字段中
  定义明确、可测量的性能目标。"NEEDS CLARIFICATION" 仅在规格创建时可接受；
  **必须**在第 1 阶段设计开始前完成确认。
- 默认性能基准（除非规格中定义了更严格的目标，否则适用）：
  - REST API 端点：在预期峰值负载下，p95 响应时间 ≤ 200 毫秒。
  - 数据库查询：单次查询执行时间 ≤ 100 毫秒；**禁止** N+1 查询问题。
  - 批处理/后台任务：**必须**在规格中定义的 SLA 窗口内完成。
  - Redis 缓存命中率：核心热点数据缓存命中率 ≥ 90%。
- 相对于上一版本性能回退 ≥ 20% 的情况**必须**视为阻塞性 Bug，**不得**发布。
- 对于引入或修改公共端点的功能，**必须**在 CI 流水线中包含验证上述目标的性能测试。
- 数据库索引：所有频繁查询字段**必须**建立索引，索引策略**必须**在 `data-model.md` 中记录。

**原因**：事后补救性能代价高昂。在规格中嵌入硬性目标可强制在设计早期进行性能考量，
并防止静默的性能退化。

---

## 质量门控

以下门控适用于 speckit 工作流的每个阶段，**必须**在 `plan.md` 的 `Constitution Check` 中验证。

| 门控项 | 时机 | 通过条件 |
|--------|------|---------|
| 可读性 | 代码审查 | 无超过 30 行的函数（除非有理由说明）；所有公共 API 已有中文文档 |
| 禁用代码生成框架 | 代码审查 | 无 Lombok、MapStruct 等注解或依赖 |
| 测试覆盖率 | 每次 PR 的 CI | 行覆盖率 ≥ 80%；无新增失败测试 |
| TDD 合规性 | 任务开始 | 测试先于实现编写且处于失败状态 |
| Bug 复现测试 | Bug 修复前 | 复现测试已编写并确认失败 |
| 性能目标已定义 | 第 1 阶段设计 | `Performance Goals` 字段已确认（无 NEEDS CLARIFICATION） |
| 性能回退检查 | 合并前 CI | 无端点相对基准回退 > 20% |
| N+1 查询检查 | 代码审查 | MyBatis SQL 无 N+1 查询问题 |
| API 响应结构 | 代码审查 | 统一使用 `code/message/data` 结构 |
| 用户体验一致性 | 规格审查 | 错误信息、输出格式和术语符合原则三 |
| 契约测试 | API 变更 PR | 契约测试套件已更新且通过 |

## 审查流程

- **代码审查**：每个 PR **必须**至少获得一名同伴的审查。审查者**必须**明确验证对全部四条
  核心原则的合规性。
- **规格审查**：在第 1 阶段设计开始前，功能规格**必须**经过审查，确认成功标准完整且
  符合用户体验一致性要求。
- **性能审查**：涉及公共端点的 PR，**必须**在 PR 描述中附上性能测试结果。
- **修正审查**：任何对本宪法提出的修正**必须**在 PR 描述中附上原因说明。修正需在合并前
  获得项目负责人的书面批准。

## 治理

本宪法优先于所有其他项目实践。当本文档与其他指南发生冲突时，以本文档为准。

修正程序：
1. 提交修改 `.specify/memory/constitution.md` 的 PR。
2. 在 PR 中说明修正类型（MAJOR / MINOR / PATCH）及原因。
3. 按语义化版本规则更新 `CONSTITUTION_VERSION`：
   - **MAJOR**：删除或重新定义某条原则。
   - **MINOR**：新增原则或章节。
   - **PATCH**：澄清、措辞调整或错别字修正。
4. 获得项目负责人的书面批准。
5. 将 `LAST_AMENDED_DATE` 更新为合并日期。
6. 将变更同步至受影响的模板（更新同步影响报告章节）。

所有 PR 和代码审查**必须**验证对上述质量门控表格的合规性。
运行时开发指导参见 `.claude/commands/speckit.*.md`。
